Lesson 01: Packaging, Imports, Tests

Learn how Python sees this repo before learning how the ML experiment runs.

Files: pyproject.toml, src/greenpoll, tests, .github/workflows/ci.yml

First Principle: Packaging Makes Code Importable

A Python file is not automatically a reusable package. Packaging tells Python where importable modules live and what dependencies must exist.

GreenPoll uses the modern src/ layout, so importable code lives under src/greenpoll.

The Build System

pyproject.toml declares setuptools as the build backend.

[build-system]
requires = ["setuptools>=68", "wheel"]
build-backend = "setuptools.build_meta"

This is the metadata pip reads when installing the project.

Project Metadata

  • name = "greenpoll": the package distribution name.
  • requires-python = ">=3.11": supported Python versions.
  • Dependencies include ultralytics, torch, pyyaml, matplotlib, and seaborn.

The dependency list tells you what this repo expects at runtime, but not whether every dependency is used in every code path.

Editable Install

The README recommends:

pip install -e .[dev]

-e means editable mode: imports point at your working tree, so Python sees source changes immediately without reinstalling.

.[dev] installs optional development tools such as pytest, pytest-cov, and ruff.

The src Layout

src/
  greenpoll/
    cli/
    plan/
    prioritize/
    detect/
    eval/
    sim/
    viz/

The src layout prevents tests from accidentally importing files just because the current directory happens to be on sys.path.

Command-Line Entry Point

pyproject.toml maps the terminal command greenpoll to greenpoll.cli:main.

[project.scripts]
greenpoll = "greenpoll.cli:main"

The current CLI is intentionally small: it parses --version and otherwise prints help.

Unit Test: Scoring

src/greenpoll/prioritize/scoring.py defines a simple target model and score.

score =
  readiness reward
  + confidence reward
  - distance penalty
  - occlusion penalty

The test checks the behavior, not just the formula: a near clear target should outrank a far occluded one.

Unit Test: Routing

src/greenpoll/plan/baselines.py implements a nearest-neighbor route heuristic.

A heuristic is a practical rule that is not guaranteed to be globally optimal. Nearest-neighbor repeatedly chooses the closest unvisited point.

The current test confirms that all points are returned and that the first chosen point is the closest to the default start.

CI: The Automated Gate

.github/workflows/ci.yml runs on pushes and pull requests.

  1. Checkout code.
  2. Set up Python 3.11.
  3. Install pip install -e .[dev].
  4. Run ruff check ..
  5. Run pytest.

What CI Does Not Prove

Passing CI does not prove GPU experiments are correct. It proves the lightweight package code passes lint and unit tests.

The training scripts depend on data, Ultralytics, GPU/MPS behavior, checkpoints, and CSV artifacts. Those are mostly integration concerns outside the fast unit tests.

Why This Matters For The Paper

Software quality and research validity overlap but are not identical.

  • Unit tests reduce bugs in small deterministic functions.
  • Experiment logs reduce ambiguity in long-running ML results.
  • Generated paper assets reduce manual transcription errors.

Key Takeaways

  • pyproject.toml defines how the project is installed, tested, and linted.
  • The installable package is currently small; the major experiment logic lives in scripts/.
  • CI is a useful safety net, but it is not a substitute for experiment reproducibility.
