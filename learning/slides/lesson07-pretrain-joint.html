Lesson 07: Joint Pretraining

How the harmonized source crops become the CrossPoll initialization.

File: scripts/pretrain_joint.py

First Principle: Transfer Learning

Transfer learning starts from weights that already encode useful visual features, then adapts them to a new task or domain.

GreenPoll uses two transfers: COCO to source-crop flowers, then source-crop flowers to kiwi.

Why Pretrain On Source Crops?

Apple, strawberry, and tomato provide more flower examples than kiwi alone.

The hypothesis is that their shared visual structure teaches a better flower feature extractor than general COCO objects alone.

CrossPoll is not zero-shot. It is source-domain pretraining followed by few-shot fine-tuning on held-out kiwi.

Inputs and Output

Input:
  data/joint/apple
  data/joint/strawberry
  data/joint/tomato

Output:
  runs/joint/pretrain/weights/best.pt

The output checkpoint is later loaded by run_baselines.py --methods crosspoll.

Device Resolution

GREENPOLL_DEVICE env var
  -> YAML training.device
  -> MPS if available
  -> CUDA fallback for Kaggle
  -> Ultralytics auto-detect

The script tries to run on Apple Silicon locally while still working on Kaggle CUDA machines.

Building The Joint data.yaml

train = [
  data/joint/apple/train/images,
  data/joint/strawberry/train/images,
  data/joint/tomato/train/images,
]
val = [
  data/joint/apple/valid/images,
  data/joint/strawberry/valid/images,
  data/joint/tomato/valid/images,
]

Ultralytics can accept lists of directories, so the script does not need to physically merge all images into one folder.

The Single-Class Contract

data = {
    "train": train,
    "val": val,
    "nc": 1,
    "names": ["flower"],
}

This is why harmonization must happen first. The pretraining script assumes all source labels now mean one class: flower.

Failure Is Intentional

If no joint train split exists, the script raises FileNotFoundError.

This is an invariant: pretraining without harmonized data is invalid, so the script should fail loudly.

Model Initialization

model = YOLO("yolov8n.pt")

This starts from COCO-pretrained YOLOv8n rather than random weights.

The scientific comparison later asks whether extra source-crop pretraining improves adaptation beyond COCO transfer alone.

Training Hyperparameters

The script reads training settings from configs/experiments/crosspoll.yaml:

  • epochs, patience, imgsz, batch.
  • optimizer, amp, cache, cos_lr, workers.
  • lr0_joint for source-domain pretraining.

Why lr0_joint Is Separate

Pretraining and fine-tuning have different optimization needs. Pretraining can use a larger learning rate because it is learning source-domain flower features from more data.

Fine-tuning later uses lr0_finetune, which is smaller to avoid destroying useful pretrained features too quickly.

Run Directory Semantics

project = runs/joint
name = pretrain
exist_ok = True

This produces a stable checkpoint path: runs/joint/pretrain/weights/best.pt.

Stable paths make downstream automation simpler, but they also mean reruns can overwrite old local artifacts.

Connection To CrossPoll

COCO yolov8n.pt
  -> joint source-crop flower training
  -> runs/joint/pretrain/weights/best.pt
  -> kiwi few-shot fine-tuning

The checkpoint is the bridge between source crops and held-out kiwi.

Key Takeaways

  • Joint pretraining is the defining step that makes CrossPoll different from ordinary COCO transfer.
  • The script depends on harmonized single-class data and a stable experiment config.
  • The saved best.pt is an experimental artifact, not a source file.
