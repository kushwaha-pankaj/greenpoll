Lesson 00: GreenPoll Map

Before reading functions, learn what this repository is trying to prove.

Goal: connect files, experiments, and the paper into one mental model.

First Principle: A Research Repo Is Evidence

A research repository is not just code. It is a chain of evidence: data choices, experiment code, logged metrics, generated figures, and paper claims.

For GreenPoll, the claim is about CrossPoll: multi-crop pretraining for flower detection, followed by few-shot adaptation to a held-out crop.

The One-Sentence Project

GreenPoll is an open-source greenhouse pollination robotics project.

CrossPoll is the current study: train a YOLOv8n flower detector on apple, strawberry, and tomato, then test how quickly it adapts to kiwi with limited labels.

The scientific question is sample efficiency: does cross-crop pretraining reduce how many kiwi labels are needed?

Big Picture Flow

Roboflow crop datasets
  -> YOLO-format crop folders
  -> harmonized source-crop labels
  -> joint pretraining checkpoint
  -> kiwi fine-tuning sweeps
  -> results/summary.csv
  -> paper table and figure

If you can trace a number backward through this chain, you understand the project at the level a reviewer will expect.

Main Directories

Research Execution

  • scripts/: download, harmonize, train, evaluate, build paper assets.
  • configs/experiments/: locked experiment settings.
  • results/summary.csv: final metric table.

Software Package

  • src/greenpoll/: installable Python package.
  • tests/: fast unit tests for small package behavior.
  • pyproject.toml: build, dependencies, pytest, Ruff.

Current Codebase Shape

The deepest behavior today is in standalone scripts, not the package modules.

  • scripts/run_baselines.py runs the main training/evaluation sweep.
  • scripts/harmonize_labels.py creates the joint source-domain data.
  • scripts/pretrain_joint.py creates the CrossPoll checkpoint.
  • scripts/build_paper_assets.py turns CSV results into paper assets.

The package modules under src/greenpoll are lightweight foundations for future robotics components.

The Experiment Contract

configs/experiments/crosspoll.yaml is the shared contract for model, crops, budgets, training parameters, metrics, and seeds.

A contract matters because one paper claim may depend on multiple scripts agreeing about the same budgets, seeds, crop split, and metric names.

Governance Files

  • ops/DECISIONS.md records why choices were made: YOLOv8n, kiwi holdout, source crops, Roboflow export, label caps.
  • ops/NEXT_ACTIONS.md and ops/SESSION_HANDOFF.md preserve continuity between work sessions.
  • CLAUDE.md captures hard-won operational constraints, especially Kaggle and paper-build gotchas.

Paper Integrity Rule

Do not hand-edit paper numbers. The paper assets are generated from results/summary.csv by scripts/build_paper_assets.py.

This protects the paper from drift: if a result changes, regenerate the table and figure from the same source file.

What To Learn First

  1. Object detection vocabulary: images, boxes, labels, IoU, mAP.
  2. YOLO data layout and label format.
  3. The experiment config and how scripts read it.
  4. The training chain from scratch/COCO/CrossPoll to summary.csv.
  5. The paper build chain from summary.csv to LaTeX assets.

Key Takeaways

  • The repo's center of gravity is the CrossPoll experiment, not a large application framework.
  • The most important invariant is traceability from paper claim back to experiment output.
  • Some README-described modules are future-facing; the lesson series will separate implemented behavior from planned behavior.
