Lesson 06: Label Harmonization

How separate crop datasets become one flower-detection source domain.

File: scripts/harmonize_labels.py

First Principle: Labels Define The Task

A model does not learn "flowers" in the abstract. It learns the relationship between pixels, bounding boxes, and numeric class IDs.

If two datasets use different class meanings, joining them without remapping can teach contradictory supervision.

YOLO Label Format

class_id x_center y_center width height
  • class_id is an integer category.
  • The four coordinates are normalized to image width and height.
  • One image can have many label lines because one image can contain many flowers.

The GreenPoll Problem

Source Crops

  • Apple: class 0 is flower.
  • Strawberry: class 0 is flower, classes 1 and 2 are fruit states.
  • Tomato: classes 0, 1, and 2 are assumed flower-like stages.

Needed Task

CrossPoll pretraining needs one shared class:

0 = flower

The Explicit Mapping

CLASS_MAP = {
    "apple":      {0: 0},
    "strawberry": {0: 0},
    "tomato":     {0: 0, 1: 0, 2: 0},
}

This is the highest-risk semantic decision in the harmonization script.

If tomato classes are not all flowers, the source-domain pretraining target is contaminated.

Dropping vs Remapping

The script uses two operations:

  • Remap: replace an original class ID with the shared class ID 0.
  • Drop: skip a line if its original class ID is not in that crop's mapping.

For strawberry, ripe and unripe fruit labels are dropped because they are not flower boxes.

Function Walk: One Label File

for line in f_in:
    parts = line.strip().split()
    if not parts:
        continue
    cls = int(parts[0])
    if cls not in mapping:
        continue
    parts[0] = str(mapping[cls])
    out_lines.append(" ".join(parts))

The coordinates are preserved exactly. Only the class ID is changed or the annotation is dropped.

Why Preserve Coordinates?

Harmonization changes label meaning, not object location. The bounding box still marks the same pixels in the same image.

If you changed coordinates here, this would become an annotation transformation script, not a class harmonization script.

Output Layout

data/joint/
  apple/train/images
  apple/train/labels
  strawberry/train/images
  strawberry/train/labels
  tomato/train/images
  tomato/train/labels

The output mirrors YOLO's expected image/label directory structure so Ultralytics can train on it.

Why Copy Images?

The code copies image folders instead of symlinking them.

The in-code reason: Ultralytics can resolve image symlinks before deriving label paths, which may cause it to read original labels instead of harmonized labels.

Copying keeps images and rewritten labels co-located under data/joint.

Idempotence

At startup, the script removes the old joint directory if it exists.

if JOINT_DIR.exists():
    shutil.rmtree(JOINT_DIR)

That makes reruns clean: stale labels from a previous mapping cannot survive.

Trade-Off: Safety vs Disk

  • Copying images avoids Ultralytics label-path bugs.
  • Copying images costs disk space.
  • This matters because the project notes limited local disk and recommends Kaggle for compute-heavy work.

Key Takeaways

  • Class harmonization is a scientific assumption, not just preprocessing.
  • Only source crops are harmonized for joint pretraining; kiwi remains held out.
  • The map in CLASS_MAP should not be changed casually because it changes what the model learns as "flower."
